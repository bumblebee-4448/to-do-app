# Full-Stack Todo List Design

## Purpose

Build a complete full-stack Todo List application from the existing README specification. The product should let users manage daily tasks through a responsive, polished, utilitarian TypeScript interface backed by a TypeScript RESTful Express API and MongoDB persistence.

## Scope

The first implementation will deliver the full README feature set:

- List todos with pagination or load-more behavior.
- Create, edit, delete, and view todos.
- Toggle todos between `pending` and `completed`.
- Search todos by title or description.
- Filter todos by status.
- Sort todos by `createdAt`, `dueDate`, or `priority` in ascending or descending order.
- Validate todo data in both client and server layers.
- Support persisted light and dark themes.
- Use optimistic updates for fast toggle, edit, and delete interactions.
- Cover core backend and frontend behavior with automated tests.

The app will not include authentication, multi-user ownership, Docker, deployment automation, or notification/reminder delivery in this implementation.

## Architecture

The repository is a two-package monorepo using TypeScript end to end and current latest package versions at implementation time:

- `server/`: Node.js, Express 5, TypeScript, MongoDB, Mongoose 9, express-validator, Jest 30, Supertest, and mongodb-memory-server.
- `client/`: Vite 8, React 19, TypeScript, React Router 7, React Query 5, Axios, Zustand 5, React Hook Form, Zod 4, Tailwind CSS 4, shadcn-style local UI components, Vitest 4, and React Testing Library.

The backend owns persistence, validation, REST status codes, response shape, and pagination metadata. The frontend owns interaction state, form state, server cache state, theme persistence, and user-facing presentation.

## Backend Design

### Data Model

`Todo` documents will contain:

- `title`: string, required, trimmed, 3-100 characters.
- `description`: string, optional, trimmed, max 500 characters.
- `priority`: enum `low`, `medium`, `high`, default `low`.
- `dueDate`: optional date.
- `status`: enum `pending`, `completed`, default `pending`.
- `createdAt` and `updatedAt`: generated by Mongoose timestamps.

### API Contract

All routes live under `/api/v1/todos`.

- `GET /`: list todos with `page`, `limit`, `search`, `status`, `sortBy`, and `order`.
- `GET /:id`: fetch one todo.
- `POST /`: create one todo.
- `PUT /:id`: replace editable todo fields.
- `PATCH /:id`: partially update editable fields, including `status`.
- `DELETE /:id`: delete one todo and return `204 No Content`.

Success responses use:

```json
{
  "success": true,
  "data": {}
}
```

List responses also include:

```json
{
  "pagination": {
    "page": 1,
    "limit": 10,
    "total": 25,
    "totalPages": 3,
    "hasNextPage": true,
    "hasPrevPage": false
  }
}
```

Error responses use:

```json
{
  "success": false,
  "message": "Validation failed",
  "errors": []
}
```

### Backend Modules

- `src/app.ts`: create and configure the Express app.
- `src/server.ts`: connect to MongoDB and start listening.
- `src/config/db.ts`: database connection helper.
- `src/models/Todo.ts`: typed Mongoose schema and model.
- `src/controllers/todoController.ts`: typed route handlers.
- `src/routes/todoRoutes.ts`: route definitions.
- `src/validations/todoValidation.ts`: express-validator rules.
- `src/middlewares/asyncHandler.ts`: async controller wrapper.
- `src/middlewares/validateRequest.ts`: convert validation errors into JSON responses.
- `src/middlewares/errorHandler.ts`: centralized error response middleware.
- `src/middlewares/notFound.ts`: 404 route fallback.
- `src/__tests__/todos.test.ts`: API tests.

## Frontend Design

### User Experience

The app opens directly to the working Todo interface, not a landing page. The layout is a responsive task workspace with:

- A compact top bar containing product title, total/open/completed counts, and theme toggle.
- A task composer for creating todos.
- Search, status filter, sort field, and sort direction controls.
- A paginated load-more todo list.
- Inline todo cards with completion checkbox, title, description, priority badge, due date, edit, and delete actions.
- Empty, loading, error, and validation states.

The visual direction follows the README: flat surfaces, thin borders, no gradients, restrained muted priority colors, responsive layout, and persisted light/dark mode.

### Client Data Flow

UI components call feature hooks. Feature hooks call API functions. API functions call the configured Axios instance.

React Query will cache todo list pages using query keys based on:

- `search`
- `status`
- `sortBy`
- `order`
- `limit`

Mutations will optimistically update the cache for create, edit, toggle, and delete. If the server request fails, the previous cache snapshot is restored and a visible error message is shown.

## Validation

Client and server validation must enforce the same business rules:

- `title`: required, trimmed, 3-100 characters.
- `description`: optional, trimmed, max 500 characters.
- `priority`: one of `low`, `medium`, `high`.
- `dueDate`: optional ISO date; when provided by the client form, it must not be in the past.
- `status`: one of `pending`, `completed`.

The server remains authoritative. Client validation improves feedback, but every API mutation is validated independently by Express middleware.

## Error Handling

The backend returns:

- `400` for validation errors and invalid request data.
- `404` for missing todos.
- `500` for unexpected server or database errors.

The frontend displays:

- Inline form errors for validation failures.
- A list-level error state when todos cannot load.
- A mutation error message when optimistic updates roll back.

## Testing Strategy

Backend tests with Jest and Supertest will cover:

- Listing todos with pagination metadata.
- Searching by keyword.
- Filtering by status.
- Creating valid todos.
- Rejecting invalid create payloads.
- Updating todos with PUT.
- Toggling status with PATCH.
- Returning `404` for missing todos.
- Deleting todos with `204`.

Frontend tests with Vitest and React Testing Library will cover:

- Rendering the main dashboard.
- Creating a todo through the form.
- Showing validation errors for invalid form data.
- Toggling a todo's completed state.
- Searching and filtering the displayed list.

## Acceptance Criteria

The implementation is complete when:

- The backend starts and exposes all documented Todo endpoints.
- MongoDB persistence works with local `MONGO_URI`.
- All success and error responses follow the documented JSON shape.
- The frontend can perform full Todo CRUD through the backend API.
- Search, status filter, sorting, and load-more pagination work from the UI.
- Client and server validation reject invalid input consistently.
- Light and dark theme persist across reloads.
- Optimistic UI updates roll back after failed mutations.
- Backend tests pass.
- Backend TypeScript typecheck and production build pass.
- Frontend tests pass.
- Frontend TypeScript typecheck passes.
- Frontend production build succeeds.
